Write semantics
  • Append-only: meeting summaries, post-feedback
  • Upsert: projects, people
  • Rewrite: user profile
Dedup

Weekly per-table dedup pass: low-confidence and recently-created entities → embed → batched LLM merge. Aliases absorb duplicates, join rows repoint, soft-delete the loser.

Design 3

Jarvis — design

3 tables: facts, entities, fact-entity mapping. Two-tier entities (domain shared, personal per-learner). End-of-topic consolidation.

Memory table
Coach_memory_facts
memory_id        uuid PK
user_id          uuid FK
convo_id         uuid FK (nullable)  -- raw transcript
type             text                -- see below
content          jsonb               -- type-specific
reinforced_count int                 -- ++ on dedup
created_at       timestamp
last_reinforced  timestamp
-- update: append-only

type values: meeting_summary, single_feedback, personal_fact.

Write semantics
  • Append-only on Coach_memory_facts and Coach_memory_entity
  • Upsert on Coach_entities (by name + aliases)
  • User profile becomes many type=personal_fact rows.
  • Append-only across all three tables
  • Updates happen only through reinforced_count on duplicate detection
  • Consolidation pass can supersede old facts ("fact Z replaces fact W; soft-delete W")
Dedup

Weekly consolidation pass: for each entity touched in the past week, look at memories tagged to it, find near-duplicates within a type, merge by incrementing reinforced_count. Scope is bounded by the entity neighborhood.

Single pass at end of every topic (~5–15 new facts). One LLM call does three things:

  • Tag new facts with existing entity IDs (or create new personal entities)
  • Merge near-duplicate facts within the entity neighborhood
  • Optionally supersede stale facts

Scope bounded by entity neighborhood: only old facts that share an entity tag with new facts are reconsidered.

Notes

What's preserved from Archive: Coach v2 (5/13)

  • convo_id stays as a column on memory rows — global join key still works
  • Forward query ("what does meeting X touch?") = filter memory by convo_id, join through mapping
  • Reverse query ("which memories touch entity Y?") = join entity → mapping → memory
  • Aliases, confidence, last_mentioned all carry over

What changes from Archive: Coach v2 (5/13)

  • User profile fields become individual memory rows, consolidated via reinforced_count instead of full rewrite
  • One mapping table replaces multiple join tables
  • Adding a new memory type or entity type is a new type value, not a new table

What's distinctive about Jarvis

  • Topic boundary as the consolidation trigger (Coach equivalent: end of meeting)
  • Domain/personal tier split — shared concepts seeded per subject; Coach doesn't need this since memory is fully personal per user
  • Slug-form canonical names for entities — makes dedup easier
  • Facts hold source array of turn IDs (Coach equivalent: convo_id on memory rows)
  • schema_version typed column for migration tracking — Coach doesn't have this

Retrieval examples (Design 2)

Two worked examples showing how to assemble memory context using the Design 2 schema: one for pre-meeting prep (calendar event known, transcript not yet), one for post-meeting coaching feedback (transcript just landed).

Example 1 — Pre-meeting context

Scenario: April is about to join a meeting with Alex and Sam, titled "CERT Creative Performance Review". She wants three things in her prep: (a) participant details (role, title); (b) past meeting summaries relevant to this meeting; (c) project candidates relevant to the participants.

Inputs available from calendar:

user_id        = u_april
participants   = ["Alex", "Sam"]
meeting_title  = "CERT Creative Performance Review"

Step 1 — Resolve participants to entity_ids

Look up the participant names in Coach_entities, matching against name and aliases:

SELECT entity_id, name, details
FROM Coach_entities
WHERE user_id = 'u_april'
  AND type    = 'person'
  AND (name IN ('Alex', 'Sam')
       OR aliases && ARRAY['Alex', 'Sam']);

Returns:

entity_id | name | details
e_008     | Alex | { role: "Data Scientist", title: "Sr.", ... }
e_018     | Sam  | { role: "Data Engineer", ... }

That covers ask #1 — participant details, directly from the entity rows.

Step 2 — Extract project entity from the meeting title

SELECT entity_id, name, details
FROM Coach_entities
WHERE user_id = 'u_april'
  AND type    = 'project'
  AND (name ILIKE '%CERT%' OR aliases && ARRAY['CERT']);

Returns e_005 (CERT Creative Performance Agent). Seed entity set: {e_008, e_018, e_005}.

Step 3 — Pull past meeting summaries relevant to this meeting

SELECT DISTINCT mf.memory_id, mf.convo_id, mf.content, mf.created_at
FROM Coach_memory_facts mf
JOIN Coach_memory_entity me ON mf.memory_id = me.memory_id
WHERE mf.user_id = 'u_april'
  AND mf.type    = 'meeting_summary'
  AND me.entity_id IN ('e_008', 'e_018', 'e_005')
ORDER BY mf.created_at DESC
LIMIT 10;

Returns the most recent ~10 meeting summaries touching Alex, Sam, or CERT. Covers ask #2.

Step 4 — Pull project candidates relevant to the participants

SELECT DISTINCT e.entity_id, e.name, e.details, COUNT(*) AS frequency
FROM Coach_entities e
JOIN Coach_memory_entity me_proj ON e.entity_id = me_proj.entity_id
JOIN Coach_memory_facts  mf      ON me_proj.memory_id = mf.memory_id
WHERE e.user_id = 'u_april'
  AND e.type    = 'project'
  AND mf.type   = 'meeting_summary'
  AND mf.memory_id IN (
    SELECT me2.memory_id
    FROM Coach_memory_entity me2
    WHERE me2.entity_id IN ('e_008', 'e_018')
  )
GROUP BY e.entity_id, e.name, e.details
ORDER BY frequency DESC
LIMIT 5;

Returns top 5 projects ranked by how often they co-occur with Alex or Sam. Covers ask #3.

Step 5 — Assemble and hand to the LLM

{
  "participants": [
    { "name": "Alex", "role": "Data Scientist", "title": "Sr." },
    { "name": "Sam",  "role": "Data Engineer" }
  ],
  "relevant_past_meetings": [
    { "date": "2026-05-04", "summary": "...", "action_items": [...] },
    { "date": "2026-04-29", "summary": "...", "action_items": [...] }
  ],
  "relevant_projects": [
    { "name": "CERT Creative Performance Agent", "status": "active", "details": "..." },
    { "name": "Coach Memory", "status": "active", "details": "..." }
  ]
}

Example 2 — Post-meeting coaching feedback

Scenario: April just finished a 1:1 with Alex about CERT (convo_005). The transcript is available and meeting-summary extraction has already run, so the meeting's entities are tagged in Coach_memory_entity. We want to generate post-meeting coaching feedback that's tailored to April using past context — past meetings on the same topics, other projects she's worked on with the same people, and her own personal facts for personalization.

Inputs:

user_id        = u_april
convo_id       = convo_005   -- the meeting that just ended
                              -- transcript already extracted; meeting_summary
                              -- row mf_010 already written, with entity tags
                              -- in Coach_memory_entity

Step 1 — Get the entities the just-finished meeting touched

The meeting summary's mapping rows already exist (written during extraction). Read them back:

SELECT e.entity_id, e.type, e.name, e.details
FROM Coach_memory_entity me
JOIN Coach_entities e ON me.entity_id = e.entity_id
WHERE me.memory_id IN (
  SELECT memory_id
  FROM Coach_memory_facts
  WHERE user_id = 'u_april'
    AND convo_id = 'convo_005'
    AND type    = 'meeting_summary'
);

Returns: e_008 (Alex), e_005 (CERT). Seed entity set for retrieval.

Step 2 — Pull related past meeting summaries (excluding this one)

SELECT DISTINCT mf.memory_id, mf.convo_id, mf.content, mf.created_at
FROM Coach_memory_facts mf
JOIN Coach_memory_entity me ON mf.memory_id = me.memory_id
WHERE mf.user_id = 'u_april'
  AND mf.type    = 'meeting_summary'
  AND mf.convo_id <> 'convo_005'     -- exclude the just-finished meeting
  AND me.entity_id IN ('e_008', 'e_005')
ORDER BY mf.created_at DESC
LIMIT 5;

Returns earlier meetings about Alex or CERT — useful background that the feedback LLM can reference (e.g., "compared to last week's CERT sync, your tone today was more direct...").

Step 3 — Pull other projects April worked on with Alex (one-hop expansion)

SELECT DISTINCT e.entity_id, e.name, e.details
FROM Coach_entities e
JOIN Coach_memory_entity me_proj ON e.entity_id = me_proj.entity_id
WHERE e.user_id = 'u_april'
  AND e.type    = 'project'
  AND e.entity_id <> 'e_005'         -- exclude the current project
  AND me_proj.memory_id IN (
    -- meetings that touched Alex
    SELECT memory_id FROM Coach_memory_entity
    WHERE entity_id = 'e_008'
  );

Returns other projects April has collaborated with Alex on. Lets feedback reference cross-project context ("Alex was also your collaborator on Coach Memory; your dynamic here mirrors that pattern...").

Step 4 — Pull personal facts about April for personalization

SELECT memory_id, content
FROM Coach_memory_facts
WHERE user_id = 'u_april'
  AND type    = 'personal_fact'
ORDER BY reinforced_count DESC
LIMIT 10;

Returns April's stable self-context (role, communication style, goals) — feedback can be tuned to her, not generic.

Step 5 — Assemble the feedback prompt context

{
  "current_meeting": {
    "convo_id": "convo_005",
    "summary": "1:1 with Alex on CERT data quality issues...",
    "action_items": ["April to check with Matti on Microsoft data",
                     "Alex to re-run pattern extraction by Friday"],
    "raw_transcript": "..."   // raw transcript also passed
  },
  "user_context": [
    "April leads the AI/ML team",
    "April values direct, evidence-based communication",
    "April has been working on memory architecture this quarter"
  ],
  "related_past_meetings_with_alex_or_cert": [
    { "date": "2026-05-04", "summary": "Earlier CERT data quality discussion...",
      "action_items": ["..."] },
    { "date": "2026-04-29", "summary": "...", "action_items": ["..."] }
  ],
  "other_projects_with_alex": [
    { "name": "Coach Memory", "status": "active",
      "details": "Memory system overhaul..." }
  ]
}

This bundle goes to the feedback-generation LLM along with the raw transcript. The LLM produces coaching feedback that knows April, references the specific people and projects involved, and connects today's behavior to longer arcs in her work.

The pattern in plain terms

Both examples follow the same four moves:

  • Seed: turn whatever you have (calendar attendees, just-finished convo_id) into a set of entity_ids.
  • Walk: join through Coach_memory_entity to find memories tagged with those entities — or join the other direction to find other entities co-occurring with those memories.
  • Rank: by recency, frequency, or entity-overlap count.
  • Hydrate: pull the full content from Coach_memory_facts or Coach_entities.

Different scenarios feed different seeds and rank by different criteria. The schema and the join shape are constant.
