Coach Memory — Three Design Comparison

April Xu · May 14, 2026 · Side-by-side view of Coach v2, Coach reshaped on Jarvis pattern, and Jarvis
This compares three live designs side by side: my current Coach v2 (typed tables), Coach reshaped to share three-table pattern (so Coach and Jarvis can run on shared infrastructure), and the actual Jarvis schema. The middle column shows what Coach v2 looks like if we collapse its typed tables into a memory / entity / mapping shape — same information, Jarvis-compatible structure.
Design 1

Coach v2 — typed tables

6 entity tables, 3 join tables. Each memory or entity type has its own schema.

Entity tables
Coach_user
user_id        uuid PK
user_profile   jsonb
version        int
updated_at     timestamp
-- update: rewrite
Coach_meeting_summary
convo_id          uuid PK, FK→transcript
user_id           uuid FK
meeting_summary   jsonb  -- {summary, action_items,
                         --  meeting_type, user_role}
created_at        timestamp
-- update: append-only
Coach_projects
proj_id          uuid PK
user_id          uuid FK
name             text
aliases          text[]
details          jsonb
confidence       text
first_mentioned  timestamp
last_mentioned   timestamp
-- update: upsert
Coach_people
person_id        uuid PK
user_id          uuid FK
name             text
aliases          text[]
details          jsonb
confidence       text
first_mentioned  timestamp
last_mentioned   timestamp
-- update: upsert
Coach_single_meeting_post_feedback
convo_id    uuid PK, FK
user_id     uuid FK
feedback    jsonb
created_at  timestamp
-- update: append-only
Coach_pattern
pattern_id      uuid PK
user_id         uuid FK
pattern         jsonb
first_observed  timestamp
last_observed   timestamp
-- update: upsert
Join tables
Coach_summary_projects
convo_id  uuid FK
proj_id   uuid FK
-- composite PK
Coach_summary_people
convo_id   uuid FK
person_id  uuid FK
-- composite PK
Coach_pattern_summaries
pattern_id  uuid FK
convo_id    uuid FK
-- composite PK
Write semantics
  • Append-only: meeting summaries, post-feedback
  • Upsert: projects, people, patterns
  • Rewrite: user profile
Dedup

Weekly per-table dedup pass: low-confidence and recently-created entities → embed → batched LLM merge. Aliases absorb duplicates, join rows repoint, soft-delete the loser.

Design 2

Coach reshaped on Jarvis pattern

3 tables: facts, entity, mapping. Type column distinguishes memory and entity variants.

Memory table
Coach_memory_facts
memory_id        uuid PK
user_id          uuid FK
convo_id         uuid FK (nullable)  -- raw transcript
type             text                -- see below
content          jsonb               -- type-specific
reinforced_count int                 -- ++ on dedup
created_at       timestamp
last_reinforced  timestamp
-- update: append-only

type values: meeting_summary, single_feedback, personal_fact (user profile fields become individual rows).

Entity table
Coach_entities
entity_id        uuid PK
user_id          uuid FK
type             text       -- see below
name             text       -- canonical
aliases          text[]
details          jsonb      -- type-specific
confidence       text
first_mentioned  timestamp
last_mentioned   timestamp
-- update: upsert (resolve by name + aliases)

type values: person, project, pattern, topic.

Mapping table
Coach_memory_entity
memory_id  uuid FK
entity_id  uuid FK
-- composite PK

Replaces Coach_summary_projects, Coach_summary_people, and Coach_pattern_summaries. One join table for all relationships.

Write semantics
  • Append-only on Coach_memory_facts and Coach_memory_entity
  • Upsert on Coach_entities (by name + aliases, like v2)
  • User profile becomes many type=personal_fact rows. Old "rewrite" semantics replaced by reinforced_count and the dedup pass.
Dedup

Weekly consolidation pass: for each entity touched in the past week, look at memories tagged to it, find near-duplicates within a type, merge by incrementing reinforced_count. Scope is bounded by the entity neighborhood — same logic as v2's per-table dedup, just unified.

What's preserved from v2
  • convo_id stays as a column on memory rows where applicable — global join key still works
  • Forward query ("what does meeting X touch?") = filter memory by convo_id, join through mapping
  • Reverse query ("which memories touch entity Y?") = join entity → mapping → memory
  • Aliases, confidence, last_mentioned all carry over
What changes from v2
  • Patterns become type=pattern entities; meetings demonstrating them are tagged via mapping (instead of having their own join table)
  • User profile fields become individual memory rows, consolidated via reinforced_count instead of full rewrite
  • One mapping table replaces three join tables
Design 3

Jarvis — design

3 tables: facts, entities, fact-entity mapping. Two-tier entities (domain shared, personal per-learner). End-of-topic consolidation.

Memory table
jarvis_memory_facts
fact_id           pk
session_id        fk
type              text  -- personal, preference,
                        --   performance, ...
content           text
reinforced_count  int
last_reinforced   timestamp
last_updated      timestamp
source            text[]  -- list of turn IDs
-- update: append-only
Entity table
jarvis_memory_entities
entity_id  pk
tier       text  -- "domain" | "personal"
name       text  -- slug form
                 --   ("pricing", "guidewire")

Two tiers: domain entities are seeded once per subject (concepts like "pricing", "customer-segmentation") and reused across all learners. personal entities are proper-noun anchors created per-learner by the LLM ("guidewire", "aaron-team").

Mapping table
jarvis_memory_fact_entities
fact_id    fk
entity_id  fk
-- composite PK
Write semantics
  • Append-only across all three tables
  • Updates happen only through reinforced_count on duplicate detection
  • Consolidation pass can supersede old facts ("fact Z replaces fact W; soft-delete W")
Dedup ("single pass for adding entities and hygiene")

Runs at the end of every topic (~5–15 new facts). Single LLM call does three things in one shot:

  • Tag the new facts with existing entity IDs (or create new personal entities)
  • Merge near-duplicate facts within the entity neighborhood
  • Optionally supersede stale facts

Scope is bounded by entity neighborhood: only old facts that share an entity tag with new facts are reconsidered. The catalog of entities itself stays small enough to review in full.

What's distinctive
  • Topic boundary as the consolidation trigger (Coach equivalent: end of meeting)
  • Domain/personal tier split — exploits shared concepts across learners (Coach doesn't really need this, since memory is fully personal per user)
  • Slug-form canonical names for entities (lowercase, hyphenated) — makes dedup easier
  • Facts hold source array of turn IDs for traceability (Coach equivalent: convo_id on memory rows)

How Design 2 maps to Design 1 (the unification, in plain terms)

Design 2 is Design 1 with the typed tables collapsed:

  • Coach_meeting_summary rows → Coach_memory_facts rows with type = "meeting_summary".
  • Coach_single_meeting_post_feedback rows → Coach_memory_facts rows with type = "single_feedback".
  • Coach_user.user_profile → multiple Coach_memory_facts rows with type = "personal_fact" (consolidated by reinforced_count instead of overwritten).
  • Coach_projects rows → Coach_entities rows with type = "project".
  • Coach_people rows → Coach_entities rows with type = "person".
  • Coach_pattern rows → Coach_entities rows with type = "pattern".
  • All three join tables (Coach_summary_projects, Coach_summary_people, Coach_pattern_summaries) → one Coach_memory_entity mapping table.

No information is lost in the move. Forward and reverse query patterns still work the same way; they just go through the mapping table instead of separate join tables. The trade-off is type-specific column structure (Design 1) versus schema flexibility plus Jarvis alignment (Design 2).
